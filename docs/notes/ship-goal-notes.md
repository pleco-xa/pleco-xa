# Ship-Goal Notes (collected during proof-of-work pass)

Per-module docs notes + repair findings gathered while planning/executing the demo proofs.

- **scripts/xa-fft.js** (Core DSP & Conversions): fft() zero-pads input to next power of 2 (output length may exceed input); stft/istft are [freq][time] librosa-format with {real,imag} objects; fft_frequencies is duplicated verbatim in xa-convert.js; recursive Cooley-Tukey with per-bin object allocation, fine for analysis but not realtime-hot-path.
- **scripts/xa-convert.js** (Core DSP & Conversions): midi_to_hz/midi_to_note reject notes outside [0,127] (librosa allows any float) and note_to_midi regex lacks double accidentals (##, bb) and unicode sharps/flats that librosa accepts; hz_to_mel defaults to Slaney formula with htk=true opt-in; fft_frequencies duplicates xa-fft's export.
- **scripts/xa-util.js** (Core DSP & Conversions): tiny() returns float32 tiny 1.1754e-38 while xa-normalize's tiny() returns 1e-10 — two conflicting definitions in the package; normalize() here takes an options object while xa-normalize's normalize() takes positional args, same name different signatures; get_fftlib/set_fftlib are browser-flavored stubs, not real backend switches.
- **scripts/xa-util.js** ⚠️: softmask() here computes X/(X+Xref)^power — diverges from librosa's X^p/(X^p+Xref^p) whenever power!=1 (xa-normalize.js has the librosa-correct implementation; this duplicate should delegate or be removed). Also show_versions() hard-codes 'librosaParity: 100%, 512/512 functions' — fictional numbers, strip before docs.
- **scripts/xa-audioio.js** (Core DSP & Conversions): load()/play()/stop() are browser-only (window.AudioContext + fetch); tone/chirp/clicks/zeroCrossings/autocorrelate/lpc/resample/toMono/mu-companding are pure and Node-safe; resample() is linear interpolation (aliases on downsample — not librosa-quality); load() with mono:false returns channels CONCATENATED flat, not 2D or interleaved; imports fft/ifft from xa-fft but never uses them (dead import).
- **scripts/xa-intervals.js** (Core DSP & Conversions): plimitIntervals implements Tenney crystal-growth (librosa-faithful) with internal memo cache; wellTemperedIntervals adds Werckmeister/Kirnberger/Young historical temperaments beyond librosa; ratiosToCents on a Float32Array input returns Float32Array (map preserves type) — cents precision is float32.
- **io/wav.js** (Core DSP & Conversions): Encode is 16-bit PCM interleaved only; decode handles PCM 16/24/32-bit int and 32-bit float with proper RIFF chunk walking (word-aligned); pure JS ArrayBuffer/DataView so it runs identically in Node, browsers, and workers — the package's designated ONE wav codec (xa-wav-encoder/audio-utils/xa-file variants delegate or are superseded).
- **scripts/xa-cache.js** (Core DSP & Conversions): Keys are fn.name + JSON.stringify(args), so Float32Array args stringify poorly (serializes as object map — large-array keys are slow and typed arrays with same content DO match, but huge keys bloat memory); get/set are O(n) in accessOrder via indexOf — fine for maxSize<=1000 default; deprecated()/moved()/rename_kw are console-warning shims for API migration, warn once per wrap.
- **scripts/xa-normalize.js** (Core DSP & Conversions): This module's softmask (|X|^p/(|X|^p+|Xref|^p)) is the librosa-correct one — prefer it over xa-util's divergent duplicate; normalize() takes positional args unlike xa-util.normalize's options object (same name, different signature — docs must disambiguate); lufs_normalize is RMS-approximated, NOT ITU-R BS.1770 K-weighted — say so loudly in docs; tiny() here is 1e-10, conflicting with xa-util's 1.17e-38.
- **scripts/debug.js** (Core DSP & Conversions): DEBUG_ENABLED is a live binding mutated by setDebug (also writes back to process.env.PLECO_DEBUG / window.PLECO_DEBUG); document PLECO_DEBUG=1 as the debug switch and keep this module out of the public API surface.
- **scripts/audio-utils.js** (Core DSP & Conversions): createLoopBuffer is browser-only; the analysis fns (computeRMS/Peak/ZCR) accept anything with numberOfChannels+getChannelData so they duck-type fine in Node; exportBufferAsWav is a thin Blob wrapper over io/wav encodeWav (the canonical codec); findZeroCrossing returns startIndex (not -1) when no crossing found — surprising sentinel, document it.
- **scripts/audio-utils.js** ⚠️: createLoopBuffer news up a live AudioContext({sampleRate}) purely to call createBuffer and never closes it — leaks a context per call and can hit browser context limits; should construct AudioBuffer directly (new AudioBuffer({...})) or use OfflineAudioContext. Not demo-blocking but fix before docs bless it.
- **scripts/xa-beat-tracker.js** (Rhythm): The ONLY beat_track/tempo exporter (librosa 0.11.0 parity, fixture-gated in tests/parity/beat.parity.test.js); tempo is lag-quantized (bins = 60*sr/(hop*k), so 120 BPM input legitimately returns 117.45); quickTempo is the explicit quick tier and never impersonates tempo(); all failure paths throw, no default BPMs.
- **scripts/xa-onset.js** (Rhythm): onset_strength is the librosa-parity envelope (parity-tested) and feeds the canonical engine; onsetDetect is a distinct fast heuristic (pickPeaks threshold = mean + absolute delta, NOT librosa peak_pick); good demo params: hopLength 512, delta 0.07.
- **scripts/xa-onset.js** ⚠️: Minor: onsetsToBeats returns a fabricated default {bpm:120} when <2 onsets (house-rule violation); onsetDetect leaves console.time noise on; computeSTFT is uncentered so onsets read up to ~93ms early vs librosa.
- **scripts/xa-bpm-algorithm.js** (Rhythm): Verbatim lb/index.html port kept for provenance (exported as the `bpm` namespace); its unique value is the progress-yielding UI-friendly pipeline + Fourier tempogram peak list — after repair, position it as the tempo-drift/stability analyzer, never the primary BPM source.
- **scripts/xa-bpm-algorithm.js** ⚠️: VERIFIED broken: estimateConstrainedTempo reduces a 4s window to ~15 energy points but its BPM search needs lags >= ~15, so the autocorrelation loop never executes and every window silently returns globalTempo — the window-by-window 'tempo stability' output is constant regardless of actual tempo changes (100to140 ramp test: global=69.84 subharmonic, all 12 windows=69.8, while its own Fourier tempogram peaks correctly read 141.3/100.9). Fix: autocorrelate the raw onset envelope per window (not the 8-bucket energy downsample) or shrink the sub-window hop. Also: estimateGlobalTempo hard-caps 70-180 BPM causing subharmonic picks on mixed-tempo material, and computeOnsetStrength/computeSimpleFFT are O(n^2) decimated DFTs (~1s for 6s of audio).
- **scripts/xa-downbeat.js** (Rhythm): Honest heuristic, hard-assumes 4/4 and a trustworthy tempo input; findMusicalLoop (the index.js export) is really a bar-aligned loop scorer and brute-forces at most the first 8 bars as loop starts.
- **scripts/xa-downbeat.js** ⚠️: Minor: phase = Math.round(beatsSinceStart % 4) can yield 4 (e.g. 3.6 rounds to 4) and those onsets are silently discarded by the phase<4 guard instead of wrapping to phase 0 — biases scoring against phase 0.
- **scripts/kick-snare-detector.js** (Rhythm): Returns null when no transient beats score threshold 5.0 — callers must handle null; the +20ms 'beat center' offset is intentional and baked in; scans only [start-100ms, start+200ms] around loop.start.
- **scripts/musical-timing.js** (Rhythm): Single pure function, already covered by tests/musical-timing.test.js; score = 0.7*whole-beat closeness + 0.3*bonus for {1,2,4,8,16}-beat divisions — note 3-beat (3/4 bar) loops are penalized by design.
- **scripts/xa-tempogram.js** (Rhythm): Do not document as librosa parity until repaired; the only fixture-gated tempogram math in the package is the private one inside xa-beat-tracker.tempo().

- **streaming/analyzers.js** (Rhythm): Already wave-verified (tests/streaming-analyzers.test.js); push() returns only newly-completed frames and read() is non-consuming; explicitly quick-tier live meters per the tier law — never a fallback for offline parity analyzers.

- **scripts/xa-rhythm.js** ⚠️: Minor: plp is a windowed-autocorrelation pulse-strength approximation, NOT librosa's Fourier-tempogram PLP — rename or note in docs; Math.max(...arr) spreads can overflow the stack on very long envelopes (>~100k frames); beat_sync drops data after the last beat boundary (emits beats.length-1 segments).

- **scripts/xa-constantq.js** ⚠️: Spot-run CONFIRMS broken: 440 Hz sine with fmin=220/24 bins peaks at bin 23 (expected 12) with a smooth monotonic magnitude ramp - output is garbage. Root causes: (1) __cqt_response multiplies TIME-domain wavelet filters (wavelet() returns windowed complex exponentials indexed by time) element-wise against STFT FREQUENCY bins - librosa FFTs the filterbank first (fft_basis); the math is category-error wrong. (2) resample is called positionally as resample(y, sr, sr_target, res_type) but xa-audioio.resample signature is (y, {origSr, targetSr}) - destructuring a number yields undefined===undefined so it silently returns y unchanged, leaving the filterbank built for sr/2 while the signal stays at sr whenever early-downsampling triggers (it does at the fmin=C1/84-bin defaults). (3) stft called with 8 args in __cqt_response/pseudo_cqt (extra dtype slot) so pad_mode receives null. (4) icqt uses a naive O(N^2) ifft_real and the same wrong basis; griffinlim_cqt inherits everything. (5) sparsity/norm/window params largely ignored (hann hardcoded in wavelet()). ~400 lines of dead private helpers (__early_downsample, __vqt_filter_fft, __num_two_factors) are never called. Effectively a rewrite against librosa constantq.py.
